# 70. 爬楼梯

> **题目链接：**[70. 爬楼梯](https://leetcode.cn/problems/climbing-stairs/)

## 题目描述

假设你正在爬楼梯。需要 `n` 阶你才能到达楼顶。
每次你可以爬 `1` 或 `2` 个台阶。你有多少种不同的方法可以爬到楼顶呢？

## 核心思考：动态规划与 Python `@cache` 记忆化

这是一道堪称计算机图形科班里的“HelloWorld”级别的经典动态规划（或者菲波那契序列）问题。

不管目标第 `n` 层楼有多高耸入云不可攀附，我们逆向倒推就会发现真相的本质：
要抵达终点第 `n` 阶：
- 唯一的可能是：你要么刚从极其危险的悬崖边沿也是就下方的 **第 `n-1` 阶跨了最后一步(1单步)** 上来。
- 要么就是刚刚正立足在低洼一点的 **第 `n-2` 阶进行了超级发力直接远眺一大跨(2双步)** 登上顶峰。

因此极其理所当然的推出核心核心状态转移方程：
`方法总数 f(n) = f(n-1) 向前走的方案 + f(n-2) 向前跨的方案`。

在 Python 极具现代化工程实践的语法特性里，我们直接用递归来实现转移最为赏心悦目。为避免极其恐怖的大量由于重叠反复去死算例如 `f(3)`这种早期基础点引发超长延时算力灾难，通过装配一枚 Python 顶配魔方武器 `@cache` 装饰器。系统就会贴心自动帮你的深度探查拦截建立无死角散列表备忘查询器缓存起来，一切水到渠成。

## 解题思路 (Python)

```python
class Solution:
    def climbStairs(self, n: int) -> int:
        @cache
        def dfs(i):
            if i <= 1:
                return 1
            return dfs(i-1) + dfs(i-2)
        
        return dfs(n)
```

## 复杂度分析

- **时间复杂度**：O(N)。在挂装搭载有 `@cache` 防御型截流外挂的担保下，尽管外观看似一棵发散爆炸递归双树分支，其实每一道 `f(x)` 层级从 1 至 N 只发生被唤醒真正调令运行至多计算一遍。
- **空间复杂度**：O(N)。主要归咎依赖来源于堆积起来隐性没被回收的长线递归潜潜深入纵向调用堆栈以及 Python 内部替你承担寄存查表的自带备忘缓存录区消耗。
