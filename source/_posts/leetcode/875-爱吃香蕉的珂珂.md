# 875. 爱吃香蕉的珂珂

> **题目链接：**[875. 爱吃香蕉的珂珂](https://leetcode.cn/problems/koko-eating-bananas/)

## 题目描述

珂珂喜欢吃香蕉。这里有 `n` 堆香蕉，第 `i` 堆中有 `piles[i]` 根香蕉。警卫已经离开了，将在 `h` 小时后回来。

珂珂可以决定她吃香蕉的速度 `k` （单位：根/小时）。每个小时，她将会选择一堆香蕉，从中吃掉 `k` 根。如果这堆香蕉少于 `k` 根，她将吃掉这堆的所有香蕉，然后这一小时内不会再吃更多的香蕉。

珂珂喜欢慢慢吃，但仍然想在警卫回来前吃掉所有的香蕉。返回她可以在 `h` 小时内吃掉所有香蕉的最小速度 `k`（`k` 为整数）。

## 核心思考：二分查找（值域二分）

这道题并非在传统的已有数组序列上进行二分查找元素，而是采用了一种极为常见且重要的思想——**对答案的值域进行二分（Binary Search on Answer）**。

之所以能够利用二分搜索寻找速度 `k`，是因为时间开销函数 $Time(speed)$ 与速度 $speed$ 之间存在严格的**单调递减关系**：速度越快，耗时越短。因此，题目转化为：寻找一个最小的整数速度 `k`，使得满足 $Time(k) \le h$。这等价于经典的“寻找满足条件的左侧边界”。

### 1. 初始版本的逻辑（求左侧边界）
在第一版实现中，我们确定了搜索的基础框架：速度下限为 `1`，上限设为一极大的固定值 `1000000000`。
通过常规的求中点进行二分，并且每次重新通过 `time_cost(piles, mid)` 判断：一旦发现当前时间耗费大于 `h`，说明速度必定慢了，需要推进 `left_speed = mid + 1`；反之如果是 `< h` 或 `== h`，表明速度可能够用也可能过快，则应当向左收缩右边界以逼近那个“最小可用速度”。

#### 解题思路 1 (基础版本 - Python)

```python
class Solution:
    def minEatingSpeed(self, piles: List[int], h: int) -> int:
        def time_cost(piles,speed):
            # 这个函数应该设计用来计算以这个效率吃所需要消耗的总时间
            times = 0
            for x in piles:
                if x % speed == 0:
                    times += (x//speed)
                else:
                    times += (x//speed + 1)
            return times

        left_speed = 1
        right_speed = 1000000000
        
        # 二分查找最靠左侧的效率
        while left_speed <= right_speed:
            mid = left_speed + (right_speed - left_speed) // 2
            if time_cost(piles,mid) > h:
                # 时间消耗比h多了，说明超时了，要加快效率
                left_speed = mid + 1
            elif time_cost(piles,mid) < h:
                right_speed = mid - 1
            elif time_cost(piles,mid) == h:
                right_speed = mid - 1
            
        return left_speed
```

初版虽然逻辑正确，但在耗时表现上却大打折扣，主要原因有几处极为明显的性能卡点：
1. **右边界过宽**：一顶多一小时吃完一堆，吃再快也不会加快速度，所以实际上最大的有效速度只需要设为数组中的大根堆数目即可（`max(piles)`），盲目设为 10 亿会白白增加无意义的二分次数。
2. **重复计算开销**：在 `while` 的分支里，最坏情况下一轮循环会去调用 3 次 `time_cost`，而这个子函数本身就是一个 $O(N)$ 级别的全数组扫瞄，代价异常昂贵。
3. **分枝运算冗余**：取天花板整数的操作完全可以用 `(x + speed - 1) // speed` 的数学等价式平替 `if-else`，以避开 Python 的算术条件分支延迟。

### 2. 极致性能优化的进阶版本
根据查明的三大痛点，对边界标位与计算机制进行极简化浓缩。引入生成器加局部缓存结果的技术进行改写。

#### 解题思路 2 (优化版 - Python)

```python
class Solution:
    def minEatingSpeed(self, piles: List[int], h: int) -> int:
        def time_cost(piles, speed):
            # 使用生成器表达式和向上取整技巧，推给底层的 C 实现来加速 sum
            return sum((x + speed - 1) // speed for x in piles)

        left_speed = 1
        right_speed = max(piles) # 缩小初始右边界

        while left_speed <= right_speed:
            mid = left_speed + (right_speed - left_speed) // 2
            
            # 将耗时缓存，避免多次 O(N) 调用
            current_cost = time_cost(piles, mid)
            
            if current_cost > h:
                left_speed = mid + 1
            else:
                # current_cost <= h，说明速度够用，尝试寻找更小的左边界
                right_speed = mid - 1

        return left_speed
```

## 复杂度分析

- **时间复杂度**：$O(N \log M)$，其中 $N$ 为 `piles` 的长度，$M$ 为 `piles` 中的最大元素。初始化时获取右边界 `max(piles)` 需要 $O(N)$，随后二分搜索域横跨范围为 $M$，因此折半总次数上限约为 $O(\log M)$。在这进行的每一轮内部折半检验中，均仅仅调用一次线性时间的扫描测算 $O(N)$。组合总耗费稳定表现为标准的常数低负载层级。
- **空间复杂度**：$O(1)$。算法利用纯量运算和底层生成器迭代结构，不挂载任何的存储列表池映射，始终严格遵守最极致的原地计算协议。
