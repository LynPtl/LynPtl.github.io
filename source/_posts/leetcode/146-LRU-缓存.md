# 146. LRU 缓存

> **题目链接：**[146. LRU 缓存](https://leetcode.cn/problems/lru-cache/)

## 题目描述

请你设计并实现一个满足  **LRU (最近最少使用)** 缓存约束的数据结构。
实现 `LRUCache` 类：
- `LRUCache(int capacity)` 以正整数作为容量实例化 LRU 缓存。
- `int get(int key)` 如果关键字 `key` 存在于缓存中，则返回关键字的值，否则返回 `-1`。
- `void put(int key, int value)` 如果关键字 `key` 已经存在，则变更其数据值 `value` ；如果不存在，则向缓存中插入该组 `key-value`。如果插入操作导致关键字数量超过 `capacity`，则应该**逐出**最久未使用的关键字。
函数 `get` 和 `put` 必须以 O(1) 的平均时间复杂度运行。

## 核心思考：OrderedDict (哈希表 + 双向链表) 的神选降临

LRU 是后端服务器开发以及硬件资源置换中极其神圣不可侵犯的基础硬核组件知识领域！
它强迫我们必须兼顾三个几乎水火不容矛盾重重的不可能三角：
1. 以 O(1) 的速度用类似字典散列索引方式查找到目标数据。（这强绑要求使用哈希表）
2. 被访问唤醒激活的数据要可以迅速移步至“防被裁员”第一阶梯的安全保险最前端。（这强绑要求链条穿插极速解缚挂钩的双向链表）
3. 末端数据超额超载发生内存警报时必须无情定点开除。（依然强绑双向长链的收尾定点爆破）

如果是纯手工死磕基础裸编，就必须乖乖撰写大量防毒代码同时操作哈希表指针和双向链表各个挂点接头 `pre` `next`。
但 Python 给出了工业级的全能瑞士军刀方案模块库神器：**`collections.OrderedDict` 库（有序化记忆散列表）**。它在底层就是一套 C 层级的“哈希双向链”合体怪物。
- 使用 `move_to_end(key, last=False)` 将刚点名读取过或是最新更新注浆操作进来的特命红人强行拽推调离抛投掷送队列前端保鲜庇护所。
- 使用 `popitem(last=True)` 一刀秒杀切断尾端冗余陈长杂质。

## 解题思路 (Python)

```python
class LRUCache:
    from collections import OrderedDict
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = OrderedDict()

    def get(self, key: int) -> int:
        if key not in self.cache.keys():
            return -1
        # 被访问过一次，立马调动提升保鲜等级调往头部免死金牌区（last=False）
        self.cache.move_to_end(key, last = False)
        return self.cache[key]

    def put(self, key: int, value: int) -> None:
        self.cache[key] = value
        # 无论新建重修都提升其生命周期调入新潮区
        self.cache.move_to_end(key, last = False)
        # 爆配驱逐边缘低俗底端流浪者清理垃圾
        if len(self.cache) > self.capacity:
            self.cache.popitem()


# Your LRUCache object will be instantiated and called as such:
# obj = LRUCache(capacity)
# param_1 = obj.get(key)
# obj.put(key,value)
```

## 复杂度分析

- **时间复杂度**：调用 `put` 与遭遇 `get` 双管齐下耗费的所有时间损效严格维持并被封框限制在严苛死板的 O(1) 等级范围。这都得益于巨轮级哈希与双向游链联动的特性发酵威效。
- **空间复杂度**：O(C)，这里 C 作为指定划分宣告核发容许的限定内存最大舱存占位总积（capacity），不会遭遇超出设定预警池底扩溢的不可控隐患故障灾难。
