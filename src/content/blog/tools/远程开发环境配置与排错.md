### <strong>文档：Windows 平台 SSH 远程开发环境配置与排错 SOP</strong>

#### <strong>1. 目标与环境</strong>

  <em> <strong>目标</strong>: 在一台Windows主机（服务端）上配置OpenSSH服务，允许另一台Windows主机（客户端）通过互联网进行安全的、基于密钥的无密码远程连接，以实现远程开发。
  </em> <strong>服务端环境</strong>:
      <em> <strong>操作系统</strong>: Windows 11
      </em> <strong>内网IP</strong>: <code>192.168.1.103</code> (通过DHCP保留实现静态分配)
      <em> <strong>账户名</strong>: <code>yan02</code>
  </em> <strong>客户端环境</strong>:
      <em> <strong>操作系统</strong>: Windows 11
      </em> <strong>账户名</strong>: <code>ptlan</code>
  <em> <strong>网络环境</strong>:
      </em> <strong>路由器</strong>: TP-Link (管理地址 <code>http://192.168.1.1</code>)
      <em> <strong>公网IP</strong>: <code>124.168.121.187</code> (示例,可通过[该网站](https://whatismyipaddress.com/)查询公网ip)
  </em> <strong>核心工具</strong>:
      <em> <code>OpenSSH for Windows</code>
      </em> <code>Visual Studio Code (Remote Development 扩展)</code>
      <em> <code>Windows PowerShell</code>

-----

#### <strong>2. 配置流程与排错记录</strong>

##### <strong>步骤 2.1: 服务端 - 配置静态内网IP (DHCP保留)</strong>

  </em> <strong>操作</strong>: 为服务端电脑分配一个固定的局域网IP地址，防止因IP地址变化导致端口转发失效。
  <em> <strong>工具</strong>: 浏览器，访问路由器管理后台。
  </em> <strong>路径</strong>: <code>Network</code> \> <code>LAN Settings</code> \> <code>Address Reservation</code>。
  <em> <strong>配置</strong>: 基于服务端网卡的MAC地址（<code>90:65:84:...</code>），添加一条地址保留规则，将其IP地址固定为 <code>192.168.1.103</code>。  

![image-compressed.png](https://pub-85d4dcece16844bf8290aa4b33608ccd.r2.dev/uploads/1773213963831-z3u2t5-image-compressed.png)

##### <strong>步骤 2.2: 服务端 - 安装与启动 OpenSSH Server</strong>

  </em> <strong>操作</strong>: 安装并启动SSH服务。
  <em> <strong>工具</strong>: <code>Windows 设置</code>，<code>Windows PowerShell (管理员)</code>。
  </em> <strong>流程</strong>:
    1.  通过 <code>设置 > 应用 > 可选功能</code> 安装 <code>OpenSSH 服务器</code>。
    2.  在管理员PowerShell中执行以下命令以启动并自启服务：
        ``<code>powershell
        Start-Service sshd
        Set-Service -Name sshd -StartupType 'Automatic'
        </code>`<code>
    3.  执行以下命令，创建防火墙入站规则以允许TCP </code>22<code> 端口的通信：
        </code>`<code>powershell
        New-NetFirewallRule -Name sshd -DisplayName 'OpenSSH Server (sshd)' -Enabled True -Direction Inbound -Protocol TCP -Action Allow -LocalPort 22
        </code>`<code>

##### <strong>步骤 2.3: 路由器 - 配置端口转发</strong>

  <em> <strong>操作</strong>: 将公网的特定端口映射至服务端的SSH端口。
  </em> <strong>工具</strong>: 浏览器，访问路由器管理后台。
  <em> <strong>路径</strong>: </code>NAT Forwarding<code> \> </code>Virtual Servers<code>。
  </em> <strong>配置</strong>: 添加新规则，具体参数如下：
      <em> <strong>Service Type</strong>: </code>VSCodeSSH<code> (自定义名称)
      </em> <strong>External Port</strong>: </code>22222<code>
      <em> <strong>Internal IP</strong>: </code>192.168.1.103<code>
      </em> <strong>Internal Port</strong>: </code>22<code>
      <em> <strong>Protocol</strong>: </code>TCP<code>
      </em> <strong>Status</strong>: </code>Enabled<code>  

![image-compressed.png](https://pub-85d4dcece16844bf8290aa4b33608ccd.r2.dev/uploads/1773213985008-lq4i48-image-compressed.png)

##### <strong>步骤 2.4: 客户端 - 首次连接与密码认证问题</strong>

  <em> <strong>操作</strong>: 客户端使用VS Code或</code>ssh<code>命令尝试连接。
  </em> <strong>命令</strong>: </code>ssh yan02@124.168.121.187 -p 22222<code>
  <em> <strong>现象</strong>: 连接时被要求输入密码，但服务端账户（</code>yan02<code>）未设置密码，仅使用PIN登录。
  </em> <strong>原因分析</strong>:
    1.  SSH协议认证基于账户<strong>密码(Password)</strong>，而非设备<strong>PIN码</strong>。
    2.  OpenSSH Server默认配置（</code>sshd_config<code>）禁止空密码账户登录。
  <em> <strong>解决方案</strong>: 在服务端，通过 </code>设置 > 账户 > 登录选项<code> 为</code>yan02<code>账户添加一个强密码。

##### <strong>步骤 2.5: 客户端/服务端 - 配置SSH密钥对</strong>

  </em> <strong>操作</strong>: 在客户端生成密钥对。

      <em> <strong>工具 (客户端)</strong>: </code>PowerShell<code>。
      </em> <strong>命令</strong>: </code>ssh-keygen -t rsa -b 4096<code> (使用默认选项)。

  <em> <strong>操作</strong>: 将公钥部署至服务端。

      </em> <strong>流程</strong>:
        1.  在客户端，复制 </code>C:\Users\ptlan\.ssh\id_rsa.pub<code> 文件内容。
        2.  在服务端，创建 </code>C:\Users\yan02\.ssh\authorized_keys<code> 文件，并将公钥内容粘贴进去。

  <em> <strong>操作</strong>: 在服务端禁用密码认证。

      </em> <strong>工具 (服务端)</strong>: </code>Notepad++ (管理员)<code>。
      <em> <strong>文件</strong>: </code>C:\ProgramData\ssh\sshd_config<code>。
      </em> <strong>修改</strong>: 将 </code>PasswordAuthentication yes<code> 修改为 </code>PasswordAuthentication no<code>（并移除行首</code>#<code>）。

##### <strong>步骤 2.6: 排错回合一 - </code>Permission Denied<code></strong>

  <em> <strong>现象</strong>: 完成密钥配置并重启</code>sshd<code>服务后，连接失败，报错 </code>Permission denied (publickey, keyboard-interactive)<code>。
  </em> <strong>诊断</strong>: 使用客户端</code>ssh -v<code>命令，日志显示公钥被服务端拒绝。问题定位在服务端。  

![image-compressed.png](https://pub-85d4dcece16844bf8290aa4b33608ccd.r2.dev/uploads/1773214011257-x4afym-image-compressed.png)

##### <strong>步骤 2.7: 排错回合二 - </code>authorized_keys<code> 文件格式与 </code>sshd_config<code> 配置</strong>


  <em> <strong>问题</strong>: </code>sshd_config<code> 文件末尾的 </code>Match Group administrators<code> 配置块，将管理员的密钥文件路径重定向至 </code>C:\ProgramData\ssh\administrators_authorized_keys<code>。  

![image-compressed.png](https://pub-85d4dcece16844bf8290aa4b33608ccd.r2.dev/uploads/1773214029341-06l0jb-image-compressed.png)

      </em> <strong>解决方案</strong>: 将该配置块的两行用 </code>#<code> 注释掉，使其失效。

##### <strong>步骤 2.8: 排错回合三 - 网络连通性验证与日志缺失</strong>

  <em> <strong>现象</strong>: 修复上述问题后，连接依然失败。同时，在</code>sshd_config<code>中设置</code>LogLevel DEBUG3<code>后，服务端的</code>事件查看器 > OpenSSH > Operational<code> 日志为空。  

![image-compressed.png](https://pub-85d4dcece16844bf8290aa4b33608ccd.r2.dev/uploads/1773214060730-ocm0gs-image-compressed.png)

  </em> <strong>诊断</strong>: 使用[在线端口检查工具](https://www.portcheckers.com/)测试公网IP的</code>22222<code>端口，结果显示<strong>Open (开启)</strong>。  

![image-compressed.png](https://pub-85d4dcece16844bf8290aa4b33608ccd.r2.dev/uploads/1773214078020-n5rslv-image-compressed.png)

  <em> <strong>结论</strong>: TCP网络路径通畅。问题是</code>sshd<code>服务在收到连接后、记入日志前即发生错误，推断为Windows文件系统ACL权限问题。

##### <strong>步骤 2.9: 最终修复 - Windows 文件系统 ACL 权限</strong>

  </em> <strong>原因分析</strong>: </code>sshd<code>服务以</code>SYSTEM<code>账户身份运行，但</code>SYSTEM<code>账户无权访问属于</code>yan02<code>用户目录下的</code>authorized_keys<code>文件。
  <em> <strong>解决方案</strong>: 在服务端，以管理员身份运行PowerShell，执行以下脚本以精确重置文件权限。
      </em> <strong>命令</strong>:
        </code>`<code>powershell
        $authorizedKeysFile = "C:\Users\yan02\.ssh\authorized_keys"
        $acl = Get-Acl $authorizedKeysFile
        $acl.SetOwner([System.Security.Principal.NTAccount]"yan02")
        $acl.Access | ForEach-Object { $acl.RemoveAccessRule($_) } | Out-Null
        $acl.SetAccessRuleProtection($true, $false)
        $acl.AddAccessRule((New-Object System.Security.AccessControl.FileSystemAccessRule("yan02", "FullControl", "Allow")))
        $acl.AddAccessRule((New-Object System.Security.AccessControl.FileSystemAccessRule("SYSTEM", "FullControl", "Allow")))
        $acl.AddAccessRule((New-Object System.Security.AccessControl.FileSystemAccessRule("Administrators", "FullControl", "Allow")))
        Set-Acl $authorizedKeysFile $acl
        </code>`<code>
  <em> <strong>操作</strong>: 重启 </code>sshd<code> 服务。
      </em> <strong>命令</strong>: </code>Restart-Service sshd<code>

##### <strong>步骤 2.10: 最终验证</strong>

  <em> <strong>操作</strong>: 客户端再次尝试连接。
  </em> <strong>结果</strong>: <strong>连接成功</strong>。实现了无需密码的密钥认证登录。

-----

#### <strong>3. 核心问题总结</strong>

本次配置过程中的关键障碍点为Windows环境下OpenSSH服务的权限问题。最终成功是基于以下关键修复：

1.  <strong>停用</code>sshd_config<code>中针对</code>administrators<code>组的</code>AuthorizedKeysFile<code>路径覆盖规则。</strong>
2.  <strong>修正</code>authorized_keys<code>文件的ACL权限</strong>，确保运行</code>sshd<code>服务的</code>SYSTEM<code>账户，拥有对用户目录下</code>authorized_keys`文件的完全控制权限。
